답은 <sos>부터
한 단어씩 잇는 끝말잇기 입니다

학습 때는 정답을 이미 알고 한 번에 채점했지만, 진짜 질문을 받으면 정답이 없습니다. 그래서 answer() 함수는 모델이 직접 고른 단어를 다시 입력 삼아 한 글자씩 답을 완성해 갑니다. 아래에서 그 과정을 직접 눌러 보며 확인해 봅시다.

1 학습 때랑 뭐가 다른가 (30초)

4단계 학습에서는 질문도, 답도 둘 다 이미 알고 있었습니다. 디코더 입력은 tgt_ids[:, :-1](<sos>부터), 정답은 tgt_ids[:, 1:](한 칸 뒤) — 정답 문장을 통째로 한 칸 밀어 넣고, 모델이 맞혔는지 비교해 손실만 줄이면 됐습니다. 이걸 티처 포싱(teacher forcing)이라 부릅니다.

그런데 지금은 새 질문 "하늘에 먹구름이 보이면 뭐가 생각나"를 받았습니다. 답을 아직 모릅니다. 정답이 없는데 dec_in을 무엇으로 만들죠? 만들 수 없습니다. 통째로 넣는 방식은 여기서 못 씁니다.

정답이 없으니, 모델이 직접 고른 단어를 다음 입력으로 재사용해서
한 칸씩 스스로 전진해야 한다 — 이것이 answer()가 존재하는 이유
🎯정리하면: 학습은 "정답을 옆에 두고 채점", 추론(생성)은 "정답 없이 한 단어씩 직접 써 내려가기"입니다. 이 둘의 차이가 오늘 살펴볼 answer() 함수의 전부입니다.

2 원리(비유): <sos>로 시작하는 끝말잇기

노트북 마크다운도 이렇게 소개합니다 — "답은 <sos>부터 한 단어씩 만듭니다. 다음 단어를 고를 땐 앞말만 볼 수 있어요. 끝말잇기랑 비슷하죠!"

진행 방식은 단순합니다. ① 지금까지 만든 답을 모델에 다시 넣고 ② 다음에 올 단어들의 점수를 계산해서 ③ 가장 점수가 높은 단어를 고르고(greedy 방식) ④ 그 단어를 답 뒤에 이어 붙입니다. ⑤ <eos>(끝 표시)가 나올 때까지 반복.

<sos>→ 비가→ 올→ 것→ 같아→ <eos> (멈춤)
🥇Greedy(그리디)란? 매 스텝마다 다음 단어 후보 전체에 점수를 매기고, logits[0, -1].argmax() — 딱 1등만 뽑는 방식입니다. 여러 후보를 저울질하지 않고 항상 "지금 가장 확실한 단어"만 선택합니다.
⚠️진짜 끝말잇기와 다른 점: 끝말잇기는 "마지막 글자"만 이어받지만, 여기서는 지금까지 만든 답 전체를 다시 읽고 다음 단어를 고릅니다. 한 글자가 아니라 전체 맥락을 매번 다시 참고하는 셈이에요.

3 왜 미래를 가리나 — 인과 마스크, 학습 vs 추론

make_causal_mask는 torch.tril(하삼각행렬)로 "미래는 못 본다"는 규칙을 만듭니다. 그런데 이 마스크가 진짜 힘을 발휘하는 순간은 학습과 추론에서 서로 다릅니다.

📚학습(교사 강요) 때: 정답 문장 전체를 한 번에 병렬로 모델에 넣습니다. 이때 마스크가 없다면 "같아"를 예측하는 자리가 바로 오른쪽에 놓인 정답 "같아" 자체를 미리 훔쳐볼 수 있어요(정답 누설). 그래서 각 위치는 자기 자신과 그 이전만 보게 강제로 가려야 합니다. 하삼각 마스크 모양을 슬라이더로 직접 만져보고 싶다면 마스킹 설명서와 디코더 블록 설명서에 이미 준비돼 있으니 먼저 보고 와도 좋습니다.

💡그런데 추론(생성) 때는 얘기가 다릅니다. answer() 안의 gen 리스트에는 이미 만들어진 단어만 들어 있습니다. 아직 태어나지 않은 미래 단어는 애초에 존재하지 않으니, make_causal_mask는 사실상 아무것도 새로 가리지 않고 조용히 통과시킵니다. 그런데도 학습·추론 양쪽에 같은 함수를 재사용하는 이유는, 같은 코드 경로로 두 상황 모두 안전하게 동작하도록 만들기 위해서입니다. 이 차이는 5번 "직접 해보기"에서 마스크 칸이 항상 전부 열려 있는 것으로 바로 확인할 수 있어요.

4 코드는 이 흐름이 전부

노트북 example-detailed.ipynb의 answer() 함수입니다 (5단계, 셀 29).

@torch.no_grad()
def answer(model, sv, tv, question, max_len=20):
    model.eval()                                    # 드롭아웃 끔 → 결과 고정
    src = torch.tensor([sv.encode(question, add_special=False)])
    src_mask = make_padding_mask(src, sv.pad_id)
    enc = model.encode(src, src_mask)               # ① 질문 인코딩, 딱 한 번만!
    gen, steps = [tv.sos_id], []                       # ② <sos>로 답 시작
    for _ in range(max_len):
        tgt = torch.tensor([gen])
        tgt_mask = make_causal_mask(tgt.size(1), tgt.device)   # ③ 인과 마스크
        logits = model.decode_step(tgt, enc, tgt_mask, src_mask)     # ④ 재계산
        nxt = logits[0, -1].argmax().item()             # ⑤ 1등만 선택(greedy)
        steps.append((tv.decode(gen), tv.itos[nxt]))
        gen.append(nxt)                                # ⑥ 답 뒤에 이어 붙임
        if nxt == tv.eos_id: break                # ⑦ 끝나면 멈춤
    tgt = torch.tensor([gen]); tgt_mask = make_causal_mask(tgt.size(1), tgt.device)
    model.decode_step(tgt, enc, tgt_mask, src_mask)     # ⑧ 완성된 답으로 한 번 더(어텐션 확보용)
    cross = model.decoder.layers[-1].cross_attn.last_attn_weights[0].mean(0)
    return steps, tv.decode(gen), cross
번호코드쉬운 말
①model.encode(src, src_mask)질문을 딱 한 번만 읽어서 기억해 둠
②gen = [sos_id]답을 <sos> 하나로 시작
③make_causal_mask지금까지의 답 안에서 미래(오른쪽)를 가림
④model.decode_step(...)지금까지의 답 전체를 다시 디코더에 통과
⑤argmax()다음 단어 후보 중 최고점 1개만 선택
⑥⑦gen.append / break답에 이어 붙이고, <eos>면 종료
⑧model.decode_step(...) 한 번 더완성된 답으로 다시 통과시켜 크로스어텐션 확보(6번에서 설명)
♻️질문(enc)은 재사용, 답(gen)은 매번 재계산. 질문 인코딩은 루프 밖에서 1번만 계산해 재사용하지만, decode_step은 매 스텝마다 지금까지의 답 전체를 처음부터 다시 디코더에 통과시킵니다. 이전 계산 결과를 저장해 두는 KV 캐시가 없는, 이해하기 쉬운 교육용 단순 구현이기 때문입니다. 실제 서비스용 LLM은 이 부분을 캐싱해서 새로 추가된 단어분만 계산합니다.

6 헷갈리기 쉬운 점: forward() vs encode()+decode_step()

Transformer 클래스(부품 ⑨)에는 경로가 두 개 있습니다. 상황에 맞는 것만 골라 씁니다.

상황쓰는 함수특징
학습model(src, tgt, ...) → forward() 정답 전체를 한 번에 병렬 처리(교사 강요)
추론(답 생성)model.encode() 1회 + model.decode_step() 반복 정답이 없어 한 단어씩 순차 생성
🔧왜 나눠 놨나: encode와 decode_step은 코드 주석 그대로 "추론 때는 질문 인코딩을 한 번만 하면 되므로 따로 떼어 둔 함수"입니다. forward() 하나로도 계산은 되지만, 매 스텝 질문을 통째로 다시 인코딩하는 낭비가 생기기 때문에 추론 전용 경로를 따로 마련해 둔 것입니다.
🔁완성 후 한 번 더 통과시키는 이유: answer() 마지막 줄은 완성된 gen을 한 번 더 decode_step에 통과시킵니다. 단어를 더 만들려는 게 아니라, 마지막 스텝 기준의 크로스어텐션 가중치를 확보해 다음 절(어텐션 히트맵)에서 시각화하기 위해서입니다.
