Mini-Transformer · 학습 다시 쉽게

학습은
"틀린 만큼 조금씩 고치기" 입니다

3단계에서 만든 부품들은 아직 텅 빈 뼈대일 뿐이에요. 예측 → 정답과 비교(loss) → 가중치 수정, 이 세 걸음을 400번 반복해야 모델이 진짜로 "먹구름 → 비"를 배웁니다. 실제 노트북을 그대로 실행해 얻은 진짜 loss 곡선으로 그 과정을 직접 훑어봅시다.

1 부품은 다 만들었는데, 왜 아직 안 되나요? (30초)

3단계에서 위치 인코딩·어텐션·인코더·디코더·마스크까지 10개 부품을 다 조립했습니다. 그런데 이 상태로 모델에 질문을 넣으면 어떻게 될까요? 아무 의미 없는 답이 나옵니다. 왜냐하면 nn.Linear, nn.Embedding 같은 부품들의 숫자(가중치)가 아직 무작위로 초기화된 값이기 때문이에요.

부품(구조)은 완성했지만, 그 부품 안의 숫자(가중치)는 아직
"먹구름 → 비"라는 규칙을 전혀 모르는 상태다
🎯학습이 하는 일: 질문을 넣어보고(예측) → 정답과 얼마나 다른지 재고(loss) → 그 차이만큼 가중치 숫자를 아주 조금씩 고쳐주는 것. 이걸 400번 반복하면, 처음엔 무작위였던 숫자들이 점점 "먹구름이 나오면 비를 가리켜라"는 규칙에 맞게 정렬됩니다.

2 비유: 눈 감고 다트 던지기 연습

학습 초반의 모델은 눈을 감고 다트를 던지는 초보와 같습니다. 처음엔 과녁 어디에 맞을지 전혀 모르죠. 그런데 던질 때마다 "얼마나 빗나갔는지"(loss)를 알려주면, 다음 던질 땐 그 정보를 참고해 팔 각도를 아주 조금 고칩니다. 이걸 수백 번 반복하면 점점 과녁 중앙 근처에 꽂히게 돼요.

① 예측

🎯 일단 던진다

지금 가진 가중치로 답을 만들어봅니다. model(...) 한 번 호출.

② 비교(loss)

📏 얼마나 빗나갔나

정답과 예측이 얼마나 다른지 숫자 하나(loss)로 잽니다. CrossEntropyLoss.

③ 수정

🔧 팔 각도 조금 고치기

빗나간 방향의 반대쪽으로 가중치를 살짝 옮깁니다. loss.backward(); opt.step().

학습 1번째 던지기 — 완전히 무작위 (과녁 밖 근처)

👀 6번 섹션에서는 이 다트가 실제 노트북을 돌려서 나온 진짜 loss 값을 따라 과녁 중앙으로 이동하는 걸 슬라이더로 직접 확인합니다.

3 예측하기 전에: 배치 만들기 (build_batches)

질문 6개·답 6개는 길이가 제각각입니다. "비가 올 것 같아"(4단어)도 있고 "아침이 밝았구나"(2단어)도 있어요. 그런데 계산은 하나의 반듯한 직사각형 표(텐서) 위에서만 됩니다. 그래서 짧은 문장 뒤를 <pad>(빈칸)로 채워 모두 같은 길이로 맞춥니다 — 마치 단체사진 찍을 때 키 작은 사람 뒤에 발판을 놓아 줄을 맞추는 것과 같아요.

노트북을 실제로 실행해 얻은 tgt_ids (답 문장 6개, 실제 값) — 회색 점선 칸이 <pad>

🔎 재미있게도 이번 데이터는 질문(src_ids) 쪽엔 패딩이 하나도 없습니다! "하늘에 OO이 보이면 뭐가 생각나" 틀이 항상 5단어로 똑같기 때문이에요. 답만 3~4단어로 길이가 달라서 패딩이 필요했습니다.
⚠️<pad>는 계산에서 아예 빠집니다. CrossEntropyLoss(ignore_index=tgt_vocab.pad_id) 덕분에 저 점선 칸들은 loss 계산에 끼어들지 않아요. 발판(패딩)은 사진엔 찍히지만 "사람 수"에는 안 들어가는 것과 같습니다.

4 티처 포싱 — 정답을 한 칸씩 미리 보여주며 연습

디코더는 "지금까지 쓴 답"을 보고 "다음 한 단어"를 맞혀야 합니다. 그런데 학습 땐 모델이 직접 만든(아직 엉망인) 답을 보게 하는 대신, 진짜 정답을 한 칸 밀어서 미리 보여줍니다. 끝말잇기 선생님이 정답을 한 글자씩 대신 불러주면서 "자, 다음 글자는?" 하고 계속 물어보는 것과 같아요. 이래야 훈련 속도가 훨씬 빠르고 안정적입니다.

실제 데이터: DATA[0] = ("...", "비가 올 것 같아") → tgt_ids[0] = [1, 13, 15, 5, 4, 2] 을 한 칸씩 어긋나게 자릅니다.

➡️ dec_in = tgt_ids[:, :-1](마지막 뺀 앞부분, <sos>부터) 를 보고, dec_tgt = tgt_ids[:, 1:](첫 칸 뺀 뒷부분, 한 칸 뒤) 를 맞히도록 훈련합니다. "<sos>"를 보면 "비가"를, "<sos> 비가"를 보면 "올"을 맞혀야 하는 식이에요.

5 코드로 보기

노트북 example-detailed.ipynb 셀 25(배치 만들기)와 셀 26(학습 루프)입니다.

# 셀 25: 길이 다른 문장들을 <pad>로 채워 직사각형 배치로
def build_batches(pairs, sv, tv):
    src = [torch.tensor(sv.encode(q, add_special=False)) for q, _ in pairs]
    tgt = [torch.tensor(tv.encode(a, add_special=True))  for _, a in pairs]
    src = pad_sequence(src, batch_first=True, padding_value=sv.pad_id)   # 길이 맞추기
    tgt = pad_sequence(tgt, batch_first=True, padding_value=tv.pad_id)
    return src, tgt
# 셀 26: (1)예측 → (2)비교 → (3)수정, 을 400번 반복
torch.manual_seed(42)                                       # 재현성(같은 결과)
src_ids, tgt_ids = build_batches(DATA, src_vocab, tgt_vocab)
model = Transformer(len(src_vocab), len(tgt_vocab), max_len=max_len)
opt  = torch.optim.Adam(model.parameters(), lr=3e-4)
crit = nn.CrossEntropyLoss(ignore_index=tgt_vocab.pad_id)     # pad는 손실에서 제외
dec_in, dec_tgt = tgt_ids[:, :-1], tgt_ids[:, 1:]         # 티처 포싱(한 칸 시프트)

for ep in range(1, 401):
    tgt_mask = make_decoder_mask(dec_in, tgt_vocab.pad_id)
    logits = model(src_ids, dec_in, src_mask, tgt_mask)                   # (1) 예측
    loss = crit(logits.reshape(-1, logits.size(-1)), dec_tgt.reshape(-1))  # (2) 비교
    opt.zero_grad(); loss.backward(); opt.step()                       # (3) 수정
코드단계쉬운 말
model(src_ids, dec_in, ...)①지금 가중치로 다음 단어들을 예측
crit(logits, dec_tgt)②예측과 정답의 차이를 숫자(loss) 하나로
loss.backward()③-1"어느 방향으로 틀렸는지" 기울기 계산
opt.step()③-2그 반대 방향으로 가중치를 한 걸음 이동

7 헷갈리기 쉬운 점

🔁데이터가 6쌍뿐인데 왜 400번이나 반복하나요? 한 번 보여주는 걸로는 모델이 패턴을 못 외웁니다. 사람도 단어 하나를 한 번 듣고 바로 외우진 못하죠. 같은 6쌍을 400번(=400 에폭) 반복해서 보여주며 조금씩 가중치를 다듬는 겁니다. 데이터가 작아 CPU로도 수 초면 끝나요.
0️⃣loss가 완전히 0이 되지는 않아요. 실제 값도 0.0063처럼 아주 작아지지만 정확히 0은 아닙니다. softmax는 확률이라 100%(=1)에 무한히 가까워질 뿐 정확히 1이 되진 않거든요. "충분히 작으면 성공"이라 체크포인트가 < 0.5라는 기준선을 둔 것입니다.
🧊src_mask는 왜 루프 밖에서 한 번만 만들까요? 질문(src_ids)은 400번 내내 바뀌지 않으니, 그 패딩 마스크도 매번 다시 계산할 필요가 없어요. 반면 tgt_mask는 루프 안에서 매번 새로 만드는데, 미래 가림(인과 마스크)이 입력 길이에 맞춰 계산되기 때문입니다.
