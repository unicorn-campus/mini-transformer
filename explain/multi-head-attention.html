Mini-Transformer · 멀티헤드 어텐션 다시 쉽게

멀티헤드 어텐션은
"카메라 여러 대로 한 장면 찍기" 입니다

어텐션(Scaled Dot-Product Attention) 하나만 쓰면 한 가지 관점만 보게 됩니다. 멀티헤드는 같은 문장을 여러 대의 카메라(헤드)로 동시에 찍어 각기 다른 관점(의미·구조·위치 등)을 모은 뒤 한 장으로 합치는 장치예요. 아래에서 헤드 수를 직접 바꿔 보며 확인해 봅시다.

1. 무슨 문제 2. 카메라 여러 대 비유 3. 쪼개기(split) 4. 왜 헤드마다 다르게 볼까 5. 직접 해보기 6. 합치기 & 코드 7. 헷갈리기 쉬운 점 8. 요약

1 어텐션 한 벌로는 부족한 이유 (30초)

이전 문서에서 배운 어텐션(Q·K·V)은 딱 한 번 계산됩니다. 질문 "먹구름이" 와 답 "비가" 의 의미 관련성은 잘 잡아내지만, 동시에 "몇 번째 단어인가", "문장 구조상 무슨 역할인가" 같은 다른 관점은 한 번의 계산에 다 담기 어려워요.

카메라 한 대로는 한 각도만 찍힌다
→ 카메라를 여러 대 두고 각자 찍은 뒤 합치자
🎥노트북 코드 주석 그대로: "한 번에 한 관점만 보지 말고, 여러 '헤드'로 나눠 서로 다른 관점(문법/의미/위치 등)에서 동시에 주목한 뒤 합칩니다. 카메라 여러 대로 같은 장면을 다른 각도에서 찍는 셈."

2 비유: 카메라 여러 대, 나중에 한 장으로

멀티헤드 어텐션이 하는 일은 딱 세 단계입니다. 이전 문서의 도서관 비유(Q=검색어·K=색인·V=내용)에 "카메라 대수" 하나만 추가하면 됩니다.

① 쪼개기

📷 카메라 배치

Q·K·V 전체를 헤드 수(h)만큼 나눠, 각 카메라가 담당할 부분(dₖ차원)을 정합니다.

② 각자 촬영

🔎 헤드별 어텐션

각 헤드가 독립적으로 이전 문서의 어텐션 공식을 그대로 계산합니다. 서로 간섭하지 않아요.

③ 합치기

🖼️ 한 장으로

헤드별 결과를 이어붙이고(merge), 선형변환(wo)으로 한 번 더 섞어 최종 결과를 냅니다.

💡핵심 제약: d_model = h × dₖ. 전체 폭(d_model)은 그대로 두고 카메라 대수(h)를 늘리면 카메라 한 대당 담당 폭(dₖ)은 줄어듭니다. 폭을 나눠 쓰는 거예요.

3 쪼개기(split) — d_model을 헤드로 나누기

실습 노트북의 체크포인트는 d_model=8 로 테스트합니다. 아래에서 헤드 수(h)를 눌러 바꾸면, 헤드 한 개당 차원 dₖ = d_model / h 이 어떻게 달라지는지 보세요.

d_model = 8 (체크포인트 셀과 동일) · 헤드 수 선택

d_model = 8 (전체 폭, 안 바뀜)dₖ = 8 / 4 = 2
📐 텐서 모양으로 보면 (B, S, d_model) → view+transpose → (B, h, S, dₖ). 헤드 축을 앞으로 빼서 헤드별로 완전히 따로 어텐션을 계산할 수 있게 만드는 것뿐, 값 자체는 그대로 재배열됩니다(정보가 사라지지 않음).
🏭실제 학습에 쓰는 진짜 값 (부품⑨ Transformer 기본 하이퍼파라미터, 실제 학습 셀에서도 그대로 사용): d_model=64, num_heads=4 → dₖ = 64 / 4 = 16. 체크포인트(8→4→2)와 똑같은 공식, 숫자만 실전 크기예요.
# split: (B, S, d_model) → (B, h, S, dk)
def split(self, x):
    b, s, _ = x.shape
    return x.view(b, s, self.h, self.dk).transpose(1, 2)   # (B, 헤드, 길이, dk)

4 헤드마다 왜 다르게 볼까?

코드를 보면 wq, wk, wv 는 각각 d_model → d_model 한 덩어리 선형변환입니다. 헤드별로 따로 만든 작은 Linear가 있는 게 아니에요. 대신 이 큰 변환의 출력을 split 으로 조각냅니다.

🧩 그런데 학습이 진행되면, wq(과 wk, wv)의 각 조각(헤드 몫)은 서로 다른 방향으로 값이 조정됩니다. 그 결과 헤드 1은 우연히 "의미 관련성"에 민감해지고, 헤드 2는 "문장 위치"에 민감해지는 식으로 자연히 역할이 나뉘는 경향이 생깁니다. 누가 정해준 게 아니라 학습이 알아서 다른 관점을 만들어내는 거예요.

아래 5번에서는 이 "관점이 다르다"는 게 실제로 어떤 모습인지, 같은 예시 벡터를 두 헤드로 쪼개 직접 계산해 보여드립니다.

5 직접 해보기 — 헤드 2개, 서로 다른 관점

이전 문서 6번 데모에서 쓴 질문 "하늘에·먹구름이·보이면·뭐가·생각나" / 답 "비가·올·것·같아" 의 4차원 예시 벡터를 그대로 가져와, 앞 2차원 = 헤드 A(날씨현상·하늘배경), 뒤 2차원 = 헤드 B(질문구조·서술) 로 쪼갠 뒤, 진짜 scaled_dot_product_attention 공식으로 헤드별 어텐션을 각각 계산합니다.

① 지금 만들 답 단어(Query) 고르기

② 비가 의 헤드 A(날씨·배경) 주목 비율

비가 의 헤드 B(구조·서술) 주목 비율

👀 "비가"를 고르면 → 헤드 A는 "먹구름이"에 강하게 쏠립니다(날씨 관련 차원만 보니까). 반면 헤드 B는 이 쿼리에서 구조/서술 차원 값이 거의 0이라 고르게 퍼집니다 — "비가"를 만들 땐 헤드 B가 별 정보를 안 주는 셈이죠. "같아"를 눌러 보면 이번엔 반대로 헤드 B가 더 뚜렷한 패턴을 보입니다. 쿼리마다 어느 헤드가 "말을 거는지"가 달라집니다.

6 합치기(merge) & 전체 코드

헤드별 결과 (B, h, S, dₖ) 를 다시 (B, S, d_model) 로 이어붙이는 게 merge, split의 역연산입니다. 그 다음 wo 선형변환이 헤드 경계를 넘어 한 번 더 정보를 섞어 줍니다.

헤드별 결과
(B,h,S,dₖ)→ merge
(B,S,d_model)→ wo(Linear)
최종 출력 
class MultiHeadAttention(nn.Module):
    def __init__(self, d_model, num_heads, dropout=0.1):
        super().__init__()
        self.h, self.dk = num_heads, d_model // num_heads      # dk = d_model / h
        self.wq = nn.Linear(d_model, d_model); self.wk = nn.Linear(d_model, d_model)
        self.wv = nn.Linear(d_model, d_model); self.wo = nn.Linear(d_model, d_model)
        self.dropout = nn.Dropout(dropout)
        self.last_attn_weights = None                 # 시각화용 보관

    def split(self, x):                                   # (B,S,d_model)→(B,h,S,dk)
        b, s, _ = x.shape
        return x.view(b, s, self.h, self.dk).transpose(1, 2)

    def merge(self, x):                                   # (B,h,S,dk)→(B,S,d_model)
        b, _, s, _ = x.shape
        return x.transpose(1, 2).contiguous().view(b, s, self.h * self.dk)

    def forward(self, q, k, v, mask=None):
        q, k, v = self.split(self.wq(q)), self.split(self.wk(k)), self.split(self.wv(v))
        ctx, attn = scaled_dot_product_attention(q, k, v, mask, self.dropout)   # 헤드별 독립 계산
        self.last_attn_weights = attn.detach()
        return self.wo(self.merge(ctx))                       # 합친 뒤 한 번 더 섞기
코드단계쉬운 말
wq/wk/wv(x)①Q·K·V를 각각 만드는 큰 선형변환 (d_model→d_model)
split(...)②헤드 수만큼 조각내 (B,h,S,dₖ)로
scaled_dot_product_attention③헤드마다 독립적으로 이전 문서의 어텐션 공식 적용
merge(ctx)④헤드 결과를 다시 (B,S,d_model)로 이어붙임
wo(...)⑤합친 결과를 한 번 더 선형변환해 최종 출력

7 헷갈리기 쉬운 점

⚖️헤드를 늘리면 무조건 좋아지나? 아니요. d_model이 고정이면 헤드(h)를 늘릴수록 헤드당 폭(dₖ)이 줄어듭니다. 헤드가 너무 많아지면 한 헤드가 담을 수 있는 정보가 너무 작아져 오히려 표현력이 떨어질 수 있어요. 이 노트북은 d_model=64에 num_heads=4(dₖ=16)로 균형을 잡았습니다.
🧪체크포인트가 실제로 확인하는 것: MultiHeadAttention(8, 4, 0.0) 에 모양 (1,3,8) 짜리 Q,K,V를 넣으면 출력도 그대로 (1,3,8) 이어야 합니다(차원 보존). 헤드로 쪼개고 다시 합치는 과정에서 모양이 안 틀어지는지를 검증하는 거예요 — 값이 아니라 "배관이 안 새는지" 확인입니다.
🔁self든 cross든 똑같이 동작합니다. 이전 문서 8번에서 본 것처럼, MultiHeadAttention도 self_attn(x,x,x) 로 부르면 셀프어텐션, cross_attn(x, enc, enc) 로 부르면 크로스어텐션이 됩니다. 헤드로 쪼개는 로직은 어느 쪽이든 동일해요.

8 30초 요약

"어텐션 한 벌은 한 관점만 본다.
그래서 d_model을 h개 헤드로 쪼개,
헤드마다 독립적으로 어텐션을 계산한 뒤,
결과를 다시 이어붙이고 wo로 섞어 하나로 합친다."
