Mini-Transformer · 마스킹 쉽게 이해하기

마스킹은
"여기는 보지 마"라는 신호입니다

트랜스포머는 문장을 통째로 한꺼번에 봅니다. 그러다 보면 곤란한 일이 두 가지 생겨요 — 길이를 맞추려고 끼워 넣은 빈칸(<pad>)까지 진짜 단어처럼 읽어 버리는 것, 그리고 답을 만들면서 아직 안 나온 미래 단어(정답)를 몰래 훔쳐보는 것. 마스킹은 이 둘을 막는 True/False 표입니다.

1. 왜 필요한가 2. 패딩 마스크 3. 인과 마스크 4. 코드 3줄 5. 직접 해보기 6. 체크포인트 7. 어디에 쓰이나 8. 요약

1 마스킹이 필요한 이유 (30초)

이 노트북은 문장 6쌍을 배치(batch)로 한 번에 학습합니다. 그런데 문장마다 길이가 달라요 — 답이 "비가 올 것 같아"(4단어)인 것도 있고 "아침이 밝았구나"(2단어)인 것도 있죠. 짧은 문장 뒤에는 <pad>를 채워 넣어 직사각형 텐서로 맞추는데, 이 빈칸을 모델이 진짜 단어처럼 취급하면 안 됩니다. 문제 ①.

그리고 디코더가 "비가"를 만드는 그 순간, 학습 데이터에는 이미 정답 "올 것 같아"가 바로 옆에 붙어 있습니다. 모델이 이걸 미리 봐 버리면 실력을 기르는 게 아니라 그냥 답을 베끼는 것이 됩니다. 문제 ②.

마스킹 = ①빈칸(pad)과 ②아직 안 쓴 미래를
True/False 표로 가려서, 모델이 정직하게 배우게 만드는 장치
🧩실제로 이 노트북엔 마스크 함수가 세 개 있어요. make_padding_mask(문제① 담당) · make_causal_mask(문제② 담당) · make_decoder_mask(둘을 합친 것, 디코더 셀프어텐션 전용). 하나씩 뜯어봅니다.

2 비유 ① 패딩 마스크 — 단체사진의 빈자리

단체사진을 찍을 때 줄마다 인원수가 다르면 보기 싫으니, 줄을 맞추려고 사람 없는 자리에 빈 의자를 놔둔다고 생각해 보세요. 나중에 사진을 볼 때 그 빈 의자를 "사람 한 명"으로 세면 이상하죠? <pad> 토큰이 바로 이 빈 의자입니다 — 길이만 맞추려고 끼운 것이지, 의미 있는 단어가 아니에요.

✅ make_padding_mask(seq, pad_id)는 각 자리가 진짜 단어(True)인지 빈 의자(False)인지 표시한 표를 돌려줍니다. 이 표를 어텐션 점수에 적용해 두면 (어텐션 문서에서 본 masked_fill(mask==0, -inf)), 빈 의자 열은 아무리 계산해도 주목 비율이 0%가 됩니다.
🕵️재밌는 사실: 이 노트북의 질문 6개는 우연히 전부 정확히 5단어예요. 그래서 질문(source) 쪽엔 사실 패딩이 하나도 생기지 않습니다! 패딩은 오히려 길이가 들쭉날쭉한 답(target) 쪽에서 생겨요. 5번 "직접 해보기"에서 실제로 확인해 봅시다.

3 비유 ② 인과 마스크 — 끝말잇기

노트북 마크다운에 이미 딱 맞는 비유가 있어요: "답은 한 단어씩 만들고, 다음 단어를 고를 땐 앞말만 볼 수 있다 — 끝말잇기랑 비슷하죠!" 끝말잇기에서 내 차례에 다음 사람이 낼 단어를 미리 알고 시작하면 반칙이듯, 디코더도 "비가"를 만드는 순간엔 뒤에 올 "올·것·같아"를 아직 몰라야 정직한 학습이 됩니다.

make_causal_mask는 하삼각(下三角) 행렬로 이걸 표현합니다. i번째 단어는 0번째부터 i번째까지만 볼 수 있고, 그 뒤(미래)는 전부 가려집니다. 슬라이더로 문장 길이를 바꿔가며 확인해 보세요.

make_causal_mask — 하삼각 마스크
✓볼 수 있음(허용) ·가려짐(차단, 미래)
📐행(row)은 "지금 만드는 단어", 열(column)은 "내가 보려는 단어"입니다. 대각선을 포함해 왼쪽 아래(자기 자신 포함, 과거)만 열려 있고, 오른쪽 위(미래)는 전부 닫혀 있어요. 이게 torch.tril(하삼각) 한 줄로 만들어지는 이유입니다.

4 코드는 이 세 함수가 전부

노트북 example-detailed.ipynb의 부품 ⑩ 마스크 만들기(셀 22)입니다. 앞서 본 비유 두 개가 그대로 함수 두 개, 그리고 그 둘을 합친 함수 하나로 들어 있어요.

# make_padding_mask: <pad> 자리를 가림. seq shape (B, S) → (B,1,1,S)
def make_padding_mask(seq, pad_id):
    return (seq != pad_id).unsqueeze(1).unsqueeze(2)              # 패딩 위치 가림

# make_causal_mask: 미래 단어를 못 보게 하는 하삼각 마스크
def make_causal_mask(seq_len, device):
    return torch.tril(torch.ones(seq_len, seq_len, device=device)).bool().unsqueeze(0).unsqueeze(0)

# make_decoder_mask: 디코더 셀프어텐션엔 두 마스크가 모두 필요 → &(논리곱)
def make_decoder_mask(tgt, pad_id):
    return make_padding_mask(tgt, pad_id) & make_causal_mask(tgt.size(1), tgt.device)
함수막는 문제모양(shape)비유
make_padding_mask① 빈칸(pad)(B,1,1,S)단체사진 빈 의자
make_causal_mask② 미래 훔쳐보기(1,1,L,L)끝말잇기 반칙
make_decoder_mask① + ②(B,1,T,T)두 규칙을 동시에 지키기
💡& 하나로 두 규칙이 합쳐지는 게 핵심입니다. 둘 다 True(볼 수 있음)여야만 최종 결과도 True — 빈칸도 아니고, 미래도 아닌 자리만 열립니다.

5 직접 해보기 — 실제 데이터로 세 마스크 비교

노트북의 실제 학습 배치에서 답(target) 문장 하나를 골라 보세요. 답마다 길이가 달라서, 일부는 <pad>가 끼어 있고 일부는 없습니다. 아래 세 그리드가 make_padding_mask · make_causal_mask · make_decoder_mask의 실제 계산 결과예요(디코더 학습 입력 dec_in 기준).

질문 → 답 쌍을 고르세요

① make_padding_mask
② make_causal_mask
③ make_decoder_mask (①&②)
✓볼 수 있음 ·가려짐 ■ 빨간 글씨 = <pad> 자리
🔍"해→아침"을 골라 보세요 — 답 "아침이 밝았구나"는 이 배치에서 가장 짧아서 <pad>가 하나 붙습니다. ①에서 그 열이 통째로 차단되고, ③(최종)에서는 그 열이 차단된 채로 + 삼각형(미래 가림)까지 겹쳐진 걸 볼 수 있어요. 반대로 "먹구름→비"는 이 배치의 최대 길이와 정확히 맞아 떨어져 pad가 전혀 없습니다 — ①이 완전히 열려 있어요.

6 체크포인트 — 하삼각의 합은 왜 6일까

노트북 체크포인트(셀 23)는 이렇게 검증합니다.

# 체크포인트: 3x3 인과 마스크의 True 개수가 1+2+3=6인지 확인
assert make_causal_mask(3, torch.device("cpu")).sum().item() == 6
print("✅ 마스크 OK")

이유는 간단합니다. 하삼각 행렬에서 i번째 행(0부터 시작)은 정확히 i+1개의 True를 갖습니다 (0번 행=1개, 1번 행=2개, 2번 행=3개…). 그래서 전체 합은 1+2+...+n = n(n+1)/2이 됩니다. n=3이면 1+2+3=6 — 위 3번 슬라이더를 3으로 놓았을 때와 정확히 같은 숫자예요.

7 어디에 쓰이나 — 인코더·디코더 지도

같은 마스크 함수라도 어디서 부르느냐에 따라 역할이 달라집니다. 실제 forward 코드 기준입니다.

위치사용하는 마스크가리는 것
인코더 셀프어텐션
EncoderLayer.forward (셀 17)		 make_padding_mask(src)질문의 <pad>만
디코더 셀프어텐션
DecoderLayer.forward · self_attn (셀 18)		 make_decoder_mask(tgt)<pad> + 미래 단어 둘 다
디코더 크로스어텐션
DecoderLayer.forward · cross_attn (셀 18)		 make_padding_mask(src)질문(K,V) 쪽 <pad>만 — 미래 개념이 없음(Q=답, K/V=질문 전체를 항상 다 봐도 됨)
⚠️헷갈리기 쉬운 점: 추론 함수 answer()(5단계, 셀 29)를 보면 tgt_mask = make_causal_mask(...)만 쓰고 make_decoder_mask는 안 씁니다! 왜냐면 추론은 문장을 한 개씩, 그것도 매 스텝 딱 필요한 길이만큼만 키워가며 만들기 때문에 애초에 <pad>가 끼어들 일이 없거든요. 패딩 가림은 여러 문장을 배치로 묶는 학습 때만 필요합니다.
